# v0.6 — Market Recorder и Replay Engine

## Назначение версии

v0.6 добавила ключевой исследовательский слой:

```text
live public WS market → file-backed recording → replay → strategy → simulated fills → replay report
```

Проблема до v0.6:

```text
Candle-level backtest из v0.3 полезен, но слишком груб для лимитных ордеров.
Он не знает реальной последовательности trades/orderbook updates внутри свечи.
```

Цель v0.6:

```text
1. Записывать реальную последовательность public WS событий.
2. Не перегружать БД high-volume tick data.
3. Переигрывать записанный рынок offline без повторного обращения к Bybit.
4. Прогонять текущую стратегию на recorded market streams.
```

## Главные архитектурные решения

### 1. File-backed raw events

Raw public WS events пишутся в файлы:

```text
JSONL
JSONL.GZ
```

PostgreSQL/SQLite хранит только metadata:

```text
recording session
file path
event count
bytes written
summary
status
```

Причина:

```text
orderbook/trade/ticker events могут быстро стать high-volume.
Писать каждое событие в relational DB на v0.6 неэффективно и усложняет retention.
```

### 2. Replay как research approximation

Replay использует реальную последовательность recorded public events, но fill model всё ещё approximation:

```text
- true queue position неизвестна;
- hidden/RPI liquidity неизвестна;
- частичные исполнения смоделированы ограниченно;
- latency/cancel delay ещё не production-grade.
```

Тем не менее replay точнее candle-level backtest, потому что использует реальный event ordering.

### 2. Market event schema

Каждая строка JSONL — отдельное событие:

```json
{
  "schema_version": "v1",
  "source": "bybit_public_ws",
  "recorded_at": "2026-06-22T05:30:00.000000+00:00",
  "exchange_ts": "2026-06-22T05:30:00.000000+00:00",
  "event_kind": "trade",
  "topic": "publicTrade.BTCUSDT",
  "symbol": "BTCUSDT",
  "sequence": 123,
  "payload": { "raw": "bybit payload" }
}
```

Поля:

```text
schema_version  версия локальной схемы event file
source          bybit_public_ws
event_kind      ticker / trade / orderbook / control / unknown
topic           original WS topic
symbol          symbol, если извлечён
sequence        локальный номер события
recorded_at     время записи локальным recorder
exchange_ts     timestamp из exchange payload, если доступен
payload         raw/normalized payload
```

### 3. Market recorder service

Файл:

```text
src/adaptive_bybit_bot/recording/recorder.py
```

Команда:

```bash
adaptive-bybit-bot record-market \
  --symbols BTCUSDT,ETHUSDT \
  --seconds 3600
```

Подписки:

```text
tickers.{symbol}
publicTrade.{symbol}
orderbook.{depth}.{symbol}
```

Особенности:

```text
- public WS only;
- no auth;
- no order placement;
- supports orderbook depth config;
- writes JSONL/JSONL.GZ;
- periodically flushes writer;
- stores session metadata in DB;
- updates service heartbeat if service-name is provided;
- chunks WS subscription args to avoid oversized subscribe payloads.
```

Настройки:

```env
MARKET_RECORDING_DIR=/data/market-recordings
MARKET_RECORDING_ORDERBOOK_DEPTH=50
MARKET_RECORDING_COMPRESS=true
MARKET_RECORDING_FLUSH_EVERY_EVENTS=1000
```

Пример имени файла:

```text
/data/market-recordings/20260622T053000Z-BTCUSDT-ETHUSDT-depth50.jsonl.gz
```

### 6. Replay engine

Файл:

```text
src/adaptive_bybit_bot/recording/replay.py
```

Команды:

```bash
adaptive-bybit-bot replay-market \
  --symbol BTCUSDT \
  --input data/market-recordings/example.jsonl.gz

adaptive-bybit-bot replay-market \
  --symbol BTCUSDT \
  --recording-id <recording-id>
```

Replay pipeline:

```text
JSONL reader
  ↓
MarketEvent stream
  ↓
LocalOrderBook reconstruction from orderbook events
  ↓
Recent trades buffer
  ↓
Trade-based candle aggregation
  ↓
FeatureEngine
  ↓
RegimeClassifier
  ↓
StrategyEngine + optional SentimentPolicy
  ↓
Replay fill simulator
  ↓
ReplayRun summary + ReplayFills persistence
```

### 7. Trade-based candle aggregation

В replay свечи строятся из recorded public trades.

Настройки:

```env
REPLAY_INTERVAL_SECONDS=60
REPLAY_EVALUATION_INTERVAL_SECONDS=10
REPLAY_WARMUP_CANDLES=60
REPLAY_TRADE_LOOKBACK_SECONDS=120
```

Смысл:

```text
REPLAY_INTERVAL_SECONDS             размер свечи
REPLAY_EVALUATION_INTERVAL_SECONDS  как часто стратегия оценивается во время replay
REPLAY_WARMUP_CANDLES               сколько свечей нужно до первых решений
REPLAY_TRADE_LOOKBACK_SECONDS       recent trades window для FeatureEngine
```

### 8. Replay fill model

Настройки:

```env
REPLAY_FILL_MODEL=trade_through
REPLAY_FORCE_CLOSE=true
```

Базовая логика:

```text
BUY intent fills when later recorded trades trade through/cross buy limit.
SELL intent fills when later recorded trades trade through/cross sell limit.
Fees are included.
Optionally force-close open position at replay end.
```

Ограничения:

```text
- no true queue modeling;
- no hidden/RPI liquidity;
- limited partial fill realism;
- no execution latency model yet;
- no real cancel/amend timing beyond local event approximation.
```

## Acceptance criteria v0.6

```text
1. record-market создаёт JSONL/JSONL.GZ файл.
2. record-market создаёт market_recording_sessions запись.
3. event_count/bytes_written обновляются при completion.
4. list-market-recordings показывает запись.
5. replay-market может прочитать JSONL/JSONL.GZ.
6. replay-market реконструирует local orderbook и recent trades.
7. replay-market создаёт market_replay_runs.
8. replay fills сохраняются в market_replay_fills.
9. API отдаёт market recordings/replays/fills.
10. Default compose не запускает recorder автоматически.
11. Все тесты v0.5 остаются зелёными.
```

## Ограничения v0.6

```text
1. Replay fill model всё ещё approximation.
2. Нет queue position modeling.
3. Нет realistic partial fills.
4. Нет latency/cancel delay model.
5. Нет стратегии отчётности по signal quality.
6. Нет retention/rotation policy для market-recordings.
7. Нет compression benchmarking.
8. Нет schema migration layer для новых таблиц.
9. Нет dashboard для replay reports.
```

## Что нельзя ломать после v0.6

```text
- Recording file format должен быть forward-compatible через schema_version.
- Replay должен читать файл streaming-режимом, не загружая весь JSONL в память.
- Recorder должен оставаться public WS only.
- Recorder не должен быть включён в default compose без явного overlay.
- DB должна хранить metadata, а не каждый raw tick.
- Replay должен работать offline по local file без Bybit connection.
```
