# v0.5 — Multi-service Podman Compose, heartbeats и strategy locks

## Назначение версии

v0.5 превратила проект из одиночного CLI/контейнера в multi-service стек.

Проблема до v0.5:

```text
У нас уже были независимые обязанности:
- API;
- WebSocket shadow strategy writer;
- instrument sync;
- FNG sync;
- paper runner;
- database.

Но они запускались в основном как отдельные CLI-команды.
```

Цель v0.5:

```text
Сделать PostgreSQL-first deployment через Podman Compose и защититься от конкурирующих signal writers.
```

## Главные архитектурные решения

### 1. PostgreSQL-first для compose

Standalone режим по-прежнему может использовать SQLite:

```env
DATABASE_URL=sqlite:////data/bot.db
```

Compose режим использует PostgreSQL:

```env
DATABASE_URL=postgresql+psycopg://bot:bot@postgres:5432/bybit_bot
```

### 2. Один strategy writer по умолчанию

В default compose stack writer:

```env
STRATEGY_WRITER_SERVICE=ws-shadow
```

Это значит:

```text
ws-shadow пишет strategy signals/intents.
bot-rest не включён в default stack.
paper-runner только симулирует fills по уже созданным intents.
api только читает.
fng-sync/instrument-sync обновляют вспомогательные справочники.
```

### 3. Soft DB locks

Добавлена таблица `strategy_locks`.

Идея:

```text
Перед записью signals/intents service берёт soft lease:
signal-writer:BTCUSDT
signal-writer:ETHUSDT
```

Если lock занят другим активным owner — service должен пропустить запись.

Это не distributed consensus, но достаточно для защиты от случайных дублей в одном Postgres-backed stack.

### 4. Service heartbeats

Добавлена таблица `service_heartbeats`.

Каждый long-running service периодически пишет:

```text
service_name
instance_id
status
started_at
last_seen_at
stale_after_seconds
details_json
```

Это даёт диагностику через CLI/API.

### 4. Новые CLI-команды v0.5

### 5. `compose.yaml`

Default services:

```text
postgres          shared state store
migrate           one-shot init-db
api               read-only HTTP API
fng-sync          periodic Alternative.me FNG cache refresh
instrument-sync   periodic Bybit instrument specs refresh
ws-shadow         public WS strategy writer
paper-runner      optional local paper-fill simulation
```

Default stack:

```bash
podman compose up --build
```

Важная настройка:

```yaml
STRATEGY_WRITER_SERVICE: ws-shadow
```

### 6. `compose.rest.yaml`

Добавлен optional override для REST polling writer:

```bash
podman compose -f compose.yaml -f compose.rest.yaml up --build postgres migrate api bot-rest
```

В override:

```env
STRATEGY_WRITER_SERVICE=bot-rest
```

Назначение:

```text
Использовать REST polling writer вместо WS writer, а не вместе с ним.
```

### 7. Compose helper scripts

Добавлены:

```text
scripts/compose-provider.sh
scripts/podman-compose-up.sh
scripts/podman-compose-down.sh
scripts/podman-compose-logs.sh
```

Смысл:

```text
- сначала попробовать podman compose;
- если provider недоступен — использовать podman-compose;

```

## Acceptance criteria v0.5

```text
1. podman compose stack содержит postgres/api/fng-sync/instrument-sync/ws-shadow/paper-runner.
2. migrate запускает init-db.
3. API запускается на 0.0.0.0:8080.
4. ws-shadow пишет heartbeat.
5. paper-runner пишет heartbeat.
6. list-services показывает running/stale services.
7. strategy lock создаётся и обновляется.
8. bot-rest не конкурирует с ws-shadow в default compose.
9. Existing SQLite standalone flow продолжает работать.
```

Историческая проверка v0.5:

```text
37 passed
```

## Ограничения v0.5

```text
1. Strategy locks — soft lease, не distributed consensus.
2. Нет Alembic migrations; schema создаётся через metadata.create_all.
3. Нет Prometheus metrics.
4. Нет structured health aggregation по latency/data freshness.
5. Market recorder ещё отсутствует до v0.6.
```

## Что нельзя ломать после v0.5

```text
- Default compose должен иметь только одного strategy writer.
- API должен оставаться read-only.
- SQLite standalone режим должен сохраняться.
- wait-db должен делать сервисы устойчивыми к разному поведению compose providers.
- fng-sync/instrument-sync не должны писать trading intents.
```
